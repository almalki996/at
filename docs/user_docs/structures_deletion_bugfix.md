# توثيق إصلاح مشكلة الحذف الجماعي (الهيكل التنظيمي) وتعارضات قاعدة البيانات

## وصف المشكلة الأساسية
عندما كان المستخدم يحاول حذف قسم رئيسي (جذر) وبداخله أقسام فرعية، كان يواجه مشكلتين رئيسيتين:
1. **خطأ 500 (Network Error / 500 Internal Server Error):** بسبب الطريقة التي يرسل بها إطار العمل `Refine` طلبات الحذف، حيث أنه يرسل طلبات متوازية (Parallel) عبر `Promise.all` لكل قسم على حدة.
2. **استجابة كاذبة وحذف مفاجئ:** بالرغم من أن الهيكل التنظيمي كان مرتبطاً بشواغر أو وظائف في جدول (`Departments`)، كان النظام يقبل الحذف دون تنبيه المستخدم.

## أسباب المشاكل فنياً (Technical Root Causes)
1. **الاستعلام المتوازي (Race Condition):**
   إرسال طلب حذف القسم الأب والقسم الفرعي في نفس الجزء من الثانية يؤدي لرفض قاعدة البيانات حذف "الأب" لأن "الابن" لم يُحذف بعد. هذا يُفشل العملية ويتسبب في بقاء عناصر معلقة (Stale UI State) في الواجهة، والتي تتطلب تحديث الصفحة `Refresh - F5` لتختفي.
2. **ضعف القيود في قاعدة بيانات Directus (ON DELETE SET NULL):**
   عند إنشاء العلاقات بين الجداول، كان إعداد Directus الافتراضي يجعل ارتباط جدول `Departments` بجدول `Structures` هو الـ `nullify`. أي أنه عند حذف الهيكل، تقوم قاعدة البيانات بوضع قيمة `Null` في الشاغر المرتبط دون إصدار رسالة خطأ تمنع الحذف.

## خطوات الحل والتطوير (The Fix)
تم اتباع نهج احترافي مزدوج لحل هذه المشكلات بشكل جذري في الواجهة الأمامية وقاعدة البيانات معاً:

### 1. تجاوز السلوك الافتراضي واستخدام (Directus Bulk Delete)
تم تعديل ملف `src/providers/data-provider.ts` لعمل _Override_ على دالة `deleteMany` المتوفرة في `Refine`.
**الكود الجديد:**
```typescript
deleteMany: async ({ resource, ids, variables, meta }: any) => {
    const response = await axiosInstance.delete(`/${resource}`, {
        baseURL: API_URL,
        data: ids // إرسال جميع الـ IDs في Body الطلب كحزمة واحدة
    });
    return { data: response.data?.data || [] };
}
```
**الفائدة:** هذا يضمن إرسال جميع المعرفات (الأب والابن) في طلب واحد ومغلف (Transaction)، مما يمنع تعارض الحذف ويجعل واجهة المستخدم دقيقة ومستجيبة في جزء من الثانية دون الحاجة لتحديث الصفحة.

### 2. فرض القيود الجبرية على مستوى الجداول (ON DELETE RESTRICT)
تم الدخول إلى حاوية قاعدة البيانات الخاصة بـ Directus وتغيير القيود على جميع الجداول المرتبطة بجدول `Departments` (وهي: `designations, job_types, Employees, Structures`).
تم تنفيذ استعلامات `SQL` والجدول `directus_relations` لتحويل:
`ON DELETE SET NULL` ➡️ `ON DELETE RESTRICT`.

**الفائدة:** الآن، إذا احتوى الشاغر الوظيفي على مسمى، نوع وظيفة، أو قطاع، فإنه يُمنع تماماً حذف القطاع أو الوظيفة من جذورهما، وتُصدر قاعدة البيانات خطأ `Constraint Violation`، والذي يلتقطه إطار الـ `Refine` بلطف ويحوله لخطأ ودي للمستخدم: 
_"فشل الحذف. يرجى التأكد من عدم وجود بيانات مرتبطة بهذا القسم"_.
