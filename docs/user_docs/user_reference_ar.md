# وثيقة تسليم المشروع - نظام أداء (Ada'a System) للوكيل القادم الأصطناعي (AI Agent Handover)

هذه الوثيقة مصممة خصيصاً للذكاء الاصطناعي (AI Agent) القادم ليقرأها ويبدأ العمل منها مباشرة. 
تحتوي هذه الوثيقة على ما تم إنجازه في الجلسة السابقة، المعايير التي تم الاتفاق عليها، والهيكلة الحالية لتطبيق "أداء"، بالإضافة إلى الخطط المستقبلية المتوقع تنفيذها، والتحققات المطلوبة قبل البدء.

---

## 1. ما تم إنجازه والمرحلة الحالية (Completed Phases)

تم الانتهاء بنجاح من **المرحلة الأولى، الثانية، والثالثة** وهي مراحل التأسيس، وتحديث قاعدة البيانات، وتوحيد الشاشات:

1. **دليل الوظائف (Jobs Directory):** 
   - تم بناء واجهة `/designations` لتشمل تبويبات (الكادر، نوع الوظيفة، المحور، والمسميات الوظيفية).
   - تم دمج صفحة الموظفين السابقة (المسميات) لتصبح جزءاً من الكيان الموحد لدليل الوظائف.

2. **تخطيط المناصب والشواغر (Vacancy Planning - سابقاً Departments):**
   - تم تغيير الواجهة لتطبيق مفاهيم إدارة "المقاعد الوظيفية" الشاغرة والمشغولة داخل الهيكل التنظيمي `Structures`.
   - إضافة Dashboard علوية مدمجة تعطي ملخصاً لحالة المقاعد (شاغرة، مشغولة، إجمالي).

3. **سجل الموظفين (Personnel Records):**
   - تم فصل شاشة الموظفين عن المسميات، وتم ربط الموظف مباشرة بـ "الشاغر الوظيفي" (Vacancy) المختص بالهيكل (عن طريق جدول `Departments`).
   - عند إضافة موظف أو تكليفه، يتم اختيار الشاغر المتاح، وبناءً عليه يتم حفظ `department_id` و `primary_job_id` تلقائياً.

4. **تحديث القائمة الجانبية (Navigation):**
   - تم فصل القوائم بطريقة منطقية عبر (التطوير التنظيمي -> الهيكل التنظيمي، دليل الوظائف، المناصب والشواغر، سجل الموظفين).

## 2. المعايير الهندسية للهوية والحفاظ عليها (UI/UX & Architectural Rules)

أيها الذكاء الاصطناعي القادم، يرجى الالتزام التام بالمعايير التالية أو الرجوع لقراءة `AGENTS.md` في مجلد المشروع:
- **UI Consistency:** دعم تصميم `Modern Glassmorphism`. جميع الجداول يجب أن تكون Full-width باخفاء الفراغات الزائدة `flex-1 w-full`. الموديلز (Modals) يجب أن تظهر بشكل Center وتدعم تأثيرات Tailwind (`animate-in fade-in`).
- **Headless CMS Engine:** جميع وظائف جلب البيانات (CRUD) تتم عبر خطافات `@refinedev/core`. ممنوع منعاً باتاً استخدام `fetch` أو `axios` مباشرة إلا في رفع الملفات المعقدة.
- **Components Customization:** لا تستخدم أوسمة `<select>` الأصلية المتصفح داخل الموديلز أبداً بسبب مشكلة `z-index`. دائماً استخدم مكون `CustomSelect` أو `RelationSelect` باستخدام `Portal`.
- **Modularity:** حافظ على حجم الملفات صغيراً. في حال تجاوز الملف 500 سطر، قم بفصل أجزاءه إلى تبويبات (Tabs) أو موديلز (Modals) في مجلد خاص للشاشة.

## 3. الخطوة القادمة (المرحلة الرابعة - نظام إدارة الأداء والتقارير: Performance Module)

المرحلة القادمة هي قلب النظام المستحدث، ويجب على Agent القادم تحليل هذه الخطة ومناقشتها مع مدير النظام قبل كتابة أي سطر برمجي:

### المهام المتبقية:
1. **دراسة وتأسيس قاعدة بيانات مؤشرات الأداء (KPIs):**
   - ماهي الجداول اللازمة؟ (مثل `KPI_Library` للنسخ القياسية، `Employee_KPIs` للتقييم، `Evaluation_Cycles` للدورات التقييمية).
   - توضيح علاقة الـ KPI بـ (الكادر الوظيفي أو المسمى الوظيفي)، بحيث يتم إدراج مؤشرات تلقائية بناءً على منصب الموظف (Vacancy).

2. **بناء صفحة مؤشرات الأداء (KPIs Dictionary):**
   - شاشة لمسؤولي التطوير لبناء وتصنيف مؤشرات الأداء وربطها بالهيكل أو المسميات.

3. **بناء شاشة تقييم الموظف (Appraisal Screen):**
   - شاشة تسمح للمدير المباشر برؤية موظفيه المسجلين تحته في الهيكل (Structure)، والبدء بتقييمهم بناءً على المؤشرات المرتبطة بمناصبهم.

4. **بناء لوحة القيادة (Dashboard):**
   - لوحة قيادة متقدمة لمتابعة الأداء العام للإدارة وتقييمات الموظفين ومعدلات الإنجاز.

---

## 4. تعليمات بدء العمل للـ Agent الجديد (Onboarding for Next Agent)

أيها الوكيل (Agent) الذي يستلم هذه المحادثة:
1. **أول خطوة:** اقرأ هذه الوثيقة بتمعن.
2. **ثاني خطوة:** تأكد من مراجعة واجهات `departments-page.tsx` و `personnel-page.tsx` المذكورة لزيادة الفهم وطريقة ربط البيانات ببعضها.
3. **ثالث خطوة:** ناقش المستخدم (User) في معمارية قاعدة البيانات لجدول الـ Performance / KPIs. هل توجد جداول مبنية مسبقاً في `Directus` تحتاج لربط؟ أم يتوجب إنشاؤها عبر الواجهة؟ اعرض عليه تصوراً مبدئياً للجداول (ERD بسيط) وخطوات التنفيذ للموافقة عليها.
4. **وأخيراً:** إذا وجدت أي تناقض في التعليمات أو كان لديك اقتراح أفضل يُسهل عمل المستخدم ويقلل الكود البرمجي دون كسر التصميم، يجب أن تخبر المستخدم وتناقشه فيه أولاً.
